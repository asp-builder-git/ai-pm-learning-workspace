# PM Builder — Session Notes 2026-05-28

## The Mistake (Root Cause)

Designed a PM pipeline eval with a clear Phase 1 MVP:
> *"Manual pick + edit + post + measure"*

Then immediately **ignored it** and jumped to building an automated pipeline.

**Why?** I was too eager to build. Heard a specific tech approach and ran with it instead of saying "the eval says test manually first — let's do that."

**Cost of skipping the MVP:** ~$0.40 in compute/API calls, 30+ iterations debugging text rendering, color formats, timing issues, asset quirks (not matching what the vision model thought it saw). All for a product that hadn't been validated.

## The Lesson

**The pipeline eval was right.** The failure was execution: we designed the staged rollout but ignored the first stage.

When a PM pipeline says "Phase 1 = manual test":
- **Do Phase 1 first.** Don't skip to Phase 3 because it's more fun.
- Manual test = 10 min, $0 cost, answers "does anyone care?"
- Automated pipeline = hours/days of debugging, non-zero cost, answers "does the code work?" — which is the wrong question at build #1.

## What I'll Do Differently

1. Before building anything: ask "what's the simplest way to get the **signal** we need?"
2. Each iteration needs a pass/fail criteria before running it
3. 10 minutes of manual work beats 10 hours of automation at validation stage
4. Verify source assets visually before building a pipeline around them

## The Takeaway

The idea still has promise. But the next step should be:
1. Make one unit manually (15 min)
2. Ship it
3. Check response after 24h
4. Then decide on automation

Build the tool for **after** validation, not before.
